<task type="auto">
  <name>Task 1: Settings UI card and pre-enrichment SF gate</name>
  <files>ui/pages/settings.py, ui/validation.py, enrichment/waterfall.py</files>
  <action>
    1. **Settings page SF card** (`ui/pages/settings.py`):
       - Add a "Salesforce Integration" section (st.subheader) BEFORE the existing provider cards section.
       - Use `st.container(border=True)` matching existing provider card pattern.
       - Three `st.text_input` fields: Username, Password (type="password"), Security Token (type="password").
       - Values pre-filled from `load_salesforce_config()`. When user changes values, update `.env` file (or show instructions to update .env manually -- follow existing pattern from settings.py for API keys).
       - "Test Connection" button: instantiates `SalesforceClient`, calls `health_check()`. On success: show `st.success(f"Connected to {org_name} ({account_count} accounts)")`. On failure: show `st.error(f"Connection failed: {error_message}")`. Catch `SalesforceAuthenticationFailed` specifically.
       - Import `SalesforceClient` from `providers.salesforce` and `load_salesforce_config` from `config.settings`.

    2. **Startup validation** (`ui/validation.py`):
       - Add SF to `validate_api_keys()` (or create a separate `validate_salesforce()` function).
       - If SF is configured (all 3 credentials non-empty), attempt `health_check()`. Report pass/fail.
       - If SF is not configured, report as "not configured" (neutral, not failure -- SF is optional).
       - This integrates with the dashboard API key status display from Phase 10.

    3. **Pre-enrichment SF gate** (`enrichment/waterfall.py`):
       - Add an optional `sf_client: Optional[SalesforceClient] = None` parameter to `WaterfallOrchestrator.__init__()`.
       - In `enrich_batch()`, BEFORE the worker loop for each chunk: if sf_client is set, collect domains from chunk rows, call `sf_client.check_domains_batch(domains)`. For each matched domain:
         a. Call `await self.db.update_company_sf_status(domain, match["sf_account_id"], match["sf_instance_url"])`
         b. Create a "skipped" `EnrichmentResult` with status "skipped_sf" and skip provider calls for that row.
       - Wrap SF calls in try/except. On ANY exception (connection error, timeout, expired session): log warning, proceed with enrichment for ALL rows in that chunk (don't block pipeline -- per user decision on SF fallback).
       - Use `asyncio.to_thread()` to call the synchronous SF client from async context (per research pitfall #5).
       - In `enrich_single()`: same pattern but for single domain.
  </action>
  <verify>
    <automated>pytest tests/test_salesforce.py -x -v && pytest tests/ -x --timeout=30</automated>
  </verify>
  <done>Settings page shows SF credentials card with working Test Connection. Pre-enrichment gate skips SF-matched companies. SF unavailability does not block enrichment. Startup validation reports SF status.</done>
</task>

<task type="auto">
  <name>Task 2: Companies page SF status column, filter, and override</name>
  <files>ui/pages/companies.py</files>
  <action>
    1. **SF Status column** in companies table:
       - Add "SF Status" column to the dataframe displayed in the companies table.
       - For companies with `sf_status == "in_sf"`: show "In SF" text with a green badge/indicator. In Streamlit st.dataframe, use column_config with a LinkColumn or custom formatting.
       - For companies without SF status: show empty cell.
       - Make the column sortable (standard st.dataframe behavior).

    2. **SF Account link**:
       - When a company has `sf_account_id` and `sf_instance_url`, construct the full URL: `{sf_instance_url}/{sf_account_id}`.
       - Display as a clickable link that opens in a new tab. Use `st.column_config.LinkColumn` for the SF Status column, where the link URL is the SF Account URL and display text is "In SF".
       - For companies not in SF, the link column shows nothing.

    3. **SF Status filter**:
       - Add an SF Status filter to the existing filter row above the companies table (alongside existing Status, Source, ICP Score filters).
       - Filter options: "All", "In SF", "Not in SF".
       - "In SF" filters to `sf_status == "in_sf"`.
       - "Not in SF" filters to `sf_status IS NULL`.
       - Default: "All" (show everything).

    4. **"Enrich Anyway" override**:
       - When companies are selected (using st.dataframe row selection) and they have `sf_status == "in_sf"`, show an "Enrich Anyway" button.
       - Clicking "Enrich Anyway" adds those companies to the enrichment queue (same session_state pattern from Phase 10-04) and navigates to the enrich page.
       - This overrides the SF skip -- the pre-enrichment gate should check a flag/list of "force enrich" domains.
       - Add a `force_enrich_domains: set[str]` to session_state that the waterfall checks: if domain is in force_enrich_domains, skip the SF gate for that domain.
  </action>
