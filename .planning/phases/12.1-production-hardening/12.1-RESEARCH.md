### Pattern 1: Simple Password Gate (SEC-01)
**What:** Check password before rendering any page content
**When to use:** Entry point of app.py, before `st.navigation()`
**Discretion recommendation:** Use a custom login form with `st.text_input(type="password")` rather than `st.secrets` alone. Store password in `st.secrets["APP_PASSWORD"]` with `os.getenv("APP_PASSWORD")` fallback. Use `st.session_state["authenticated"]` to persist login across reruns within a session.

```python
# Source: Streamlit docs + CONTEXT.md decision
import streamlit as st
import os
import hmac

def _get_app_password() -> str:
    """Get app password from secrets.toml or environment variable."""
    try:
        return st.secrets["APP_PASSWORD"]
    except (KeyError, FileNotFoundError):
        return os.getenv("APP_PASSWORD", "")

def check_password() -> bool:
    """Show login form, return True if authenticated."""
    if st.session_state.get("authenticated"):
        return True

    password = _get_app_password()
    if not password:
        # No password configured -- allow access with warning
        st.warning("No APP_PASSWORD configured. Set it in .streamlit/secrets.toml or as environment variable.")
        return True

    st.markdown("### Clay-Dupe Login")
    entered = st.text_input("Password", type="password", key="login_password")
    if st.button("Log in", type="primary"):
        if hmac.compare_digest(entered, password):
            st.session_state["authenticated"] = True
            st.rerun()
        else:
            st.error("Incorrect password")
    return False

# In app.py, before st.navigation():
if not check_password():
    st.stop()
```

**Key details:**
- `hmac.compare_digest()` prevents timing attacks (stdlib, no dependency)
- `st.session_state["authenticated"]` persists across page navigations within a session
- Fallback chain: `st.secrets["APP_PASSWORD"]` -> `os.getenv("APP_PASSWORD")` -> warn and allow access
- No cookie persistence across browser restarts (acceptable for shared-password model)
- Place auth check BEFORE `st.set_page_config()` is not possible (Streamlit requires it first), so auth goes after page config but before navigation

### Pattern 2: SOQL Escaping (SEC-02)
**What:** Escape user-supplied domain values before embedding in SOQL LIKE clauses
**When to use:** Any dynamic SOQL query construction

**SOQL escaping rules** (from Salesforce official docs):
- Single quotes: `'` -> `\'`
- Backslashes: `\` -> `\\`
- For LIKE clauses, also escape wildcards: `%` -> `\%`, `_` -> `\_`
- Strip or reject null bytes and other control characters

```python
# Source: Salesforce SOQL injection prevention docs
import re

def _escape_soql(value: str) -> str:
    """Escape a string for safe use in SOQL queries.

    Handles single quotes, backslashes, and LIKE wildcards.
    """
    # Remove control characters (null bytes, etc.)
    value = re.sub(r'[\x00-\x1f]', '', value)
    # Escape backslashes first (before adding more backslashes)
    value = value.replace('\\', '\\\\')
    # Escape single quotes
    value = value.replace("'", "\\'")
    # Escape LIKE wildcards
    value = value.replace('%', '\\%')
    value = value.replace('_', '\\_')
    return value

# Usage in check_domains_batch():
like_clauses = " OR ".join(
    f"Website LIKE '%{_escape_soql(d)}%'" for d in chunk
)
```

**Confidence: HIGH** -- Salesforce official docs explicitly recommend `String.escapeSingleQuotes()` in Apex; the Python equivalent is manual escaping since simple-salesforce does not provide a built-in escaping utility.

### Pattern 4: Settings Persistence via set_key (OPS-01)
**What:** Write settings changes back to .env file
**When to use:** Settings page Save button

```python
# Source: python-dotenv docs
from dotenv import set_key, load_dotenv
from pathlib import Path

ENV_PATH = Path(__file__).parent.parent / ".env"

def persist_setting(key: str, value: str) -> None:
    """Write a single key-value pair to .env file."""
    set_key(str(ENV_PATH), key, value)

def persist_settings(settings: dict[str, str]) -> None:
    """Write multiple settings to .env file, then reload."""
    for key, value in settings.items():
        set_key(str(ENV_PATH), key, value)
    # Reload environment so os.getenv() reflects new values
    load_dotenv(str(ENV_PATH), override=True)
```

**Important caveat:** `set_key()` modifies the .env file in-place but `os.getenv()` caches values. After calling `set_key()`, you MUST call `load_dotenv(override=True)` to refresh `os.environ`. The existing `reload_api_keys()` function already calls `load_dotenv()` -- verify it uses `override=True`.

**Docker consideration:** The .env file is mounted via `env_file:` in docker-compose.yml. The file at `./data/.env` or the project root `.env` must be writable by the container user. Since we're adding a non-root USER directive (OPS-02), ensure the .env file is owned by or writable to that user.

### Anti-Patterns to Avoid
- **Don't use `streamlit-authenticator` for this phase** -- the CONTEXT.md decision is a simple shared password, not per-user auth with bcrypt/YAML. The detailed auth research is useful background but the decision is simpler.
- **Don't create an `app_settings` SQLite table for this phase** -- the CONTEXT.md decision is `.env` persistence via `set_key()`. The hybrid SQLite approach from SETTINGS_PERSISTENCE.md is deferred.
- **Don't add health check endpoint** -- explicitly deferred in CONTEXT.md.
- **Don't try to make SOQL use bind variables** -- SOQL (Salesforce Object Query Language) does NOT support bind variables when queried via REST API / simple-salesforce. Escaping is the correct approach.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Timing-safe string comparison | Custom comparison loop | `hmac.compare_digest()` | Stdlib, constant-time, prevents timing attacks |
| .env file writing | Custom file parser/writer | `python-dotenv set_key()` | Already a dependency, handles quoting/escaping |
| SOQL escaping | Copy-paste Apex `String.escapeSingleQuotes` | Custom `_escape_soql()` (see pattern above) | No Python library exists; but the logic is ~5 lines, well-defined |

## Common Pitfalls

### Pitfall 1: st.set_page_config Must Be First
**What goes wrong:** Placing auth check before `st.set_page_config()` causes Streamlit error
**Why it happens:** Streamlit requires `st.set_page_config()` as the first Streamlit command
**How to avoid:** Place auth gate AFTER `st.set_page_config()` but BEFORE `st.navigation()`
**Warning signs:** `StreamlitAPIException: set_page_config() can only be called once per app`

### Pitfall 2: os.getenv Stale After set_key
**What goes wrong:** After `set_key()` writes to .env, `os.getenv()` still returns old value
**Why it happens:** `set_key()` writes to the file but doesn't update `os.environ`
**How to avoid:** Always call `load_dotenv(override=True)` after `set_key()` calls
**Warning signs:** Settings appear saved but app still uses old values until restart

### Pitfall 3: SOQL LIKE Escaping Order
**What goes wrong:** Escaping backslashes after escaping quotes produces double-escaped strings
**Why it happens:** Each escape operation adds backslashes that the next operation then escapes
**How to avoid:** Always escape backslashes FIRST, then single quotes, then LIKE wildcards
**Warning signs:** SOQL queries fail with "malformed query" or return no results

### Pitfall 4: Docker Non-Root User File Permissions
**What goes wrong:** Non-root user can't write to /data or .env
**Why it happens:** Files created by root during build are not writable by the app user
**How to avoid:** `chown` the data directory to the app user in the Dockerfile, and ensure .env is writable
**Warning signs:** `PermissionError` on first write to database or .env

### Pitfall 5: Dockerfile CMD Flags Already Clean
**What goes wrong:** Searching for `--server.enableCORS=false` in Dockerfile and not finding it
**Why it happens:** Looking at the current Dockerfile, the CORS/XSRF disable flags are NOT present in the CMD. The audit may have been referencing a previous version or docker-compose override.
**How to avoid:** Verify the actual current state of Dockerfile and docker-compose.yml before making changes. The current Dockerfile CMD is clean. The current docker-compose.yml has no command override.
**Warning signs:** Unnecessary changes to files that are already correct

### Pitfall 6: Auth Bypass on Direct Page URL
**What goes wrong:** User navigates directly to `/pages/settings.py` URL and bypasses auth
**Why it happens:** Auth check only in main app.py, not in individual pages
**How to avoid:** The auth gate in `app.py` before `st.navigation()` handles this -- `st.stop()` prevents ANY page from rendering. Verify this works with Streamlit's `_is_entry_point()` guard.

## Open Questions

1. **Docker .env writability**
   - What we know: `set_key()` needs write access to .env file
   - What's unclear: In Docker, .env is loaded via `env_file:` directive which reads at container start. The file on disk may not be mounted writable.
   - Recommendation: Mount .env as a volume (`- ./.env:/app/.env`) in docker-compose.yml so changes persist on host. Alternatively, mount just the data dir and copy .env there on first run.

2. **Budget limits in .env format**
   - What we know: Budget limits are per-provider (daily + monthly), and .env only supports strings
   - What's unclear: Best key naming convention for per-provider budgets
   - Recommendation: Use `BUDGET_DAILY_APOLLO=500`, `BUDGET_MONTHLY_APOLLO=10000` pattern. Parse in `load_settings()`.

3. **Provider enabled/disabled persistence**
   - What we know: CONTEXT.md says "provider toggles also persisted"
   - What's unclear: How to handle `enabled` field -- currently not in .env at all
   - Recommendation: Add `{PROVIDER}_ENABLED=true/false` keys to .env. Default to `true` if key not present.
